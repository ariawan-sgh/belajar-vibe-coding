# 2. Dasar Arsitektur

## 2.1 Clean Architecture

Backend menerapkan Clean Architecture dengan pemisahan lapisan yang jelas:

- **Handler Layer**: Menerima HTTP request, melakukan validation, dan memetakan ke service
- **Service Layer**: Business logic, orchestrasi use case, dan enforcement aturan bisnis
- **Repository Layer**: Abstraksi akses data, hanya operasi CRUD dan query spesifik
- **Entity Layer**: Domain model murni tanpa dependency eksternal
- **Middleware**: Cross-cutting concern (auth, logging, telemetry)

Frontend menerapkan arsitektur berbasis komponen dengan pemisahan:
- **Pages**: Komponen halaman penuh
- **Components**: Komponen UI reusable
- **Stores**: State management terpusat
- **Services**: Klien API dan utilitas
- **Types**: Defini TypeScript dan domain model

## 2.2 SOLID Principles

| Prinsip | Penerangan | Penerapan |
|---------|-----------|-----------|
| Single Responsibility | Setiap modul atau kelas memiliki satu alasan untuk berubah | Service hanya berisi business logic; Repository hanya akses data; Handler hanya mapping request/response |
| Open/Closed | Terbuka untuk ekstensi, tertutup untuk modifikasi | Gunakan interface dan dependency injection untuk memperluas perilaku tanpa mengubah kode inti |
| Liskov Substitution | Subtype dapat menggantikan base type | Interface Repository dapat digunakan oleh implementasi apapun tanpa mengubah handler |
| Interface Segregation | Pisahkan interface besar menjadi interface kecil khusus | Jangan buat repository interface dengan method yang tidak digunakan oleh service terkait |
| Dependency Inversion | Bergantung pada abstraksi, bukan konkreti | Service menerima Repository interface, bukan struct database konkret |

## 2.3 Domain-Driven Design (DDD)

- **Entity**: Struct dengan identity unik (`user_id`, `outlet_id`, `analysis_id`)
- **Value Object**: Struktur tanpa identity yangćŹŹčż°çš„ nilai (`Coordinate`, `LayerConfig`, `AnalysisParameter`)
- **Repository Interface**: Kontrak akses data per aggregate root (`UserRepository`, `OutletRepository`, `AnalysisRepository`)
- **Domain Service**: Logic yang tidak cocok pada satu entity (`CoordinateValidator`, `AnalysisRunner`, `ImportProcessor`)
- **Domain Events**: Event internal untuk perubahan state (`UserCreated`, `ImportCompleted`, `AnalysisFinished`)

# 3. Coding Standard

## 3.1 Backend (Golang)

- Mengikuti **Effective Go** dan **Go Code Review Comments**
- Format kode menggunakan `gofmt` (standar Go)
- Nama file menggunakan **snake_case** (`auth_handler.go`, `outlet_service.go`)
- Package name menggunakan **lowercase single word** atau **lowercase_snake** untuk sub-package (`service`, `repository`, `handler`, `domain/auth`)
- Error handling menggunakan `fmt.Errorf` dengan wrapping: `fmt.Errorf("failed to validate outlet: %w", err)`
- Menggunakan context untuk semua operasi I/O: `ctx context.Context`
- Mengembalikan error daripada panic untuk kondisi yang diharapkan
- Menggunakan typed error atau error code untuk kesalahan bisnis
- Menggunakan `uuid` untuk primary key alih-alih auto-increment integer

### 3.1.1 Constraint atau Enum

Semua enum atau constraint ditentukan menggunakan typed constant:

```go
type OutletStatus string

const (
    OutletStatusAktif   OutletStatus = "AKTIF"
    OutletStatusTutup   OutletStatus = "TUTUP"
    OutletStatusRenovasi OutletStatus = "RENOVASI"
)
```

Validasi menggunakan switch atau helper method. Jangan menerima string mentah untuk field enum.

### 3.1.2 Database Query

- Menggunakan parameterized query untuk mencegah SQL injection
- Query spasial menggunakan PostGIS function (`ST_DWithin`, `ST_Buffer`, `ST_Transform`)
- Menggunakan spatial index pada kolom `geom`
- Menggunakan `pgx` atau driver PostgreSQL yang mendukung PostGIS

## 3.2 Frontend (TypeScript + Solid JS)

- Mengikuti **TypeScript strict mode**
- Menggunakan interface untuk type definition, bukan type alias untuk object shape
- Component menggunakan PascalCase (`LoginPage`, `OutletForm`, `GISMapWorkspace`)
- File menggunakan PascalCase untuk component dan camelCase untuk util (`useAuth.ts`, `apiClient.ts`)
- Setiap komponen hanya mengekspor satu default atau named export utama
- State management menggunakan reactive primitive Solid JS (`createSignal`, `createStore`)
- Side effect menggunakan `createEffect` atau `onMount`
- Event handler menggunakan handler pattern (`onSubmit`, `onClick`) tanpa inline function jika memungkinkan

## 3.3 SQL dan Migration

- Nama tabel menggunakan **snake_case** plural (`outlets`, `competitors`, `analysis_sessions`)
- Nama kolom menggunakan **snake_case** (`user_id`, `latitude`, `geom`)
- Foreign key menggunakan pola `<table_singular>_id` (`user_id`, `outlet_id`, `analysis_id`)
- Migration dibuat dalam file terurut dengan timestamp prefix
- Setiap migration idempoten dan reversible jika memungkinkan

# 6. Testing Standard

## 6.1 Backend Testing

- Menggunakan **testify** untuk assertion dan mocking
- Menggunakan **testcontainers** untuk integration test dengan PostgreSQL dan PostGIS
- File test menggunakan suffix `_test.go`
- Setiap handler, service, dan repository WAJIB memiliki unit test
- Integration test WAJIB menguji alur end-to-end dari handler ke repository

### 6.1.1 Unit Test

- Meniru dependency repository dan eksternal service menggunakan mock
- Fokus pada business logic dan branching kondisi
- Coverage minimum **85%** untuk service dan handler
- Nama test menggunakan format `Test<FuncName>_<Scenario>_<ExpectedResult>`

```go
func TestAuthService_Login_ValidCredentials_ReturnsToken(t *testing.T) {
    // arrange
    // act
    // assert
}
```

### 6.1.2 Integration Test

- Menjalankan PostgreSQL dengan PostGIS menggunakan testcontainers
- Menjalankan API handler lengkap tanpa mock database
- Menggunakan database test terpisah yang di-drop setelah test
- Menutupi skenario validasi, error handling, dan happy flow
- Nama test menggunakan prefix `TestIntegration_`

### 6.1.3 Test Data

- Seed data menggunakan fixture atau factory function
- Data test menggunakan UUID tetap atau factory untuk reproducibility
- Tidak ada hardcoded credential; menggunakan environment variable atau konfigurasi test

## 6.2 Frontend Testing

- Menggunakan **Vitest** untuk unit test komponen
- Menggunakan **Testing Library** untuk render dan interaksi komponen
- Menggunakan **Playwright** untuk E2E test jika diperlukan
- Setiap komponen yang berisi logic harus memiliki unit test
- Melakukan snapshot test untuk komponen UI yang stabil jika diperlukan
- Menghindari testing implementation detail; fokus pada user behavior

# 7. Security Standard

## 7.1 Credential dan Secret Management

- **DIKETAHUI**: Tidak boleh hardcode credential, API key, token, atau password
- Semua credential menggunakan environment variable atau secret manager
- File konfigurasi `.env.example` disediakan tanpa nilai rahasia
- File `.env` dan `.env.local` dicantumkan pada `.gitignore`
- Credential tidak boleh dicetak ke log
- Password disimpan sebagai `password_hash` menggunakan algoritma bcrypt atau argon2

## 7.2 Authentication dan Authorization

- Menggunakan JWT Bearer Token untuk seluruh endpoint terprotected
- Token harus memiliki expiry time yang masuk akal
- Logout menghapus token dari sisi client dan dapat dilakukan blacklist token jika diperlukan
- RBAC menggunakan role `Admin`, `Manager`, `Analyst` sesuai dengan API contract
- Middleware autentikasi WAJIB dijalankan sebelum handler
- Setiap endpoint yang membutuhkan otorisasi khusus divalidasi di RBAC middleware atau service

## 7.3 Input Validation

- Semua input diverifikasi di handler layer sebelum proses lebih lanjut
- Validasi menggunakan validator terstruktur (`go-playground/validator` atau sejenisnya)
- Field yang bersifat enum hanya menerima nilai yang telah ditentukan
- Coordinate harus berada pada range WGS84: latitude `-90` sampai `90`, longitude `-180` sampai `180`
- File upload hanya menerima format yang diizinkan (`CSV`, `Excel`) dengan batas ukuran file
- Response error tidak boleh mengembalikan informasi internal seperti stack trace atau detail database

### 7.3.1 Validation Rules

| Field | Rule |
|-------|------|
| `username` | Required, alphanumeric, minimum 3 karakter |
| `email` | Required, format email valid, unique |
| `password` | Required, minimum 8 karakter |
| `latitude` | Required, range `-90` sampai `90` |
| `longitude` | Required, range `-180` sampai `180` |
| `status` (outlet) | Required, enum `AKTIF`, `TUTUP`, `RENOVASI` |

## 7.4 Error Handling

Semua error harus terstruktur dan konsisten sesuai API contract:

```go
type AppError struct {
    Code    string      `json:"code"`
    Message string      `json:"message"`
    Details interface{} `json:"details,omitempty"`
}
```

- Handler menangkap error dari service dan memetakan ke HTTP status code
- Service mengembalikan error dengan context yang jelas menggunakan error code
- Repository mengembalikan error tertentu untuk `not found`, `conflict`, dan `internal`
- Tidak ada panic yang membocorkan ke client; panic hanya digunakan untuk bug programmer

### 7.4.1 Error Code Mapping

| HTTP Status | Error Code | Penggunaan |
|-------------|------------|-----------|
| 400 | `BAD_REQUEST` | Validasi gagal, input tidak sesuai format |
| 401 | `UNAUTHORIZED` | Token tidak valid atau tidak ada |
| 403 | `FORBIDDEN` | Role tidak memiliki akses |
| 404 | `NOT_FOUND` | Data yang diminta tidak ditemukan |
| 409 | `CONFLICT` | Duplikasi data (misal email atau username sudah ada) |
| 422 | `UNPROCESSABLE_ENTITY` | Request valid secara sintaks tetapi gagal secara semantik |
| 500 | `INTERNAL_ERROR` | Kesalahan server yang tidak diharapkan |

# 8. Logging Standard

- Menggunakan structured logging (JSON atau structured text)
- Semua request masuk harus memiliki request ID
- Semua response keluar harus mencatatkan status code, latency, dan user ID jika terautentikasi
- Error harus dicatatkan dengan level `error` atau `warn` sesuai severity
- Tidak boleh mencetak credential, token, password, atau data sensitif
- Format log minimal: `timestamp`, `level`, `service`, `message`, `request_id`, `user_id`, `duration_ms`
- Interval log untuk operasi async (import, analysis) harus mencatatkan progress

---

# 9. OpenTelemetry Standard

- Semua service backend WAJIB menginstrumentasikan OpenTelemetry
- Melakukan tracing untuk seluruh request HTTP
- Melakukan tracing untuk operasi database dan file storage
- Melakukan tracing untuk proses async seperti import dan analysis
- Semua span harus memiliki atribut: `http.method`, `http.route`, `http.status_code`, `user.role`
- Error span harus menambahkan atribut `error.type` dan `error.message`
- Metric wajib mencakup: request count, request duration, error count per endpoint
- Instrumentasi diinisialisasi di `cmd/server/main.go` dan diekspor ke collector yang Sesuai
- Trace context harus dipropagasi ke downstream service dan repository

# 10. Git Flow dan Conventional Commit

## 10.1 Branch Strategy

Mengikuti **Git Flow** dengan branch-type:

| Branch | Purpose |
|--------|---------|
| `main` | Kode production yang stabil |
| `develop` | Kode pengembangan aktif sebelum rilis |
| `feature/<name>` | Fitur baru (dikelompokkan per modul atau per item backlog) |
| `bugfix/<name>` | Perbaikan bug pada `develop` |
| `hotfix/<name>` | Perbaikan bug darurat pada `main` |
| `release/<version>` | Persiapan rilis dari `develop` ke `main` |

Setiap feature atau bugfix dibuat dari `develop` dan di-merge kembali ke `develop`. Rilis dibuat dari `develop` ke `main`.

## 10.2 Conventional Commit Format

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

### 10.2.1 Jenis Commit

| Type | Penggunaan |
|------|-----------|
| `feat` | Fitur baru yang sesuai requirement |
| `fix` | Perbaikan bug |
| `docs` | Perubahan dokumentasi |
| `refactor` | Perubahan kode tanpa perubahan perilaku |
| `test` | Penambahan atau perbaikan test |
| `chore` | Maintenance, dependency, atau build script |
| `perf` | Peningkatan performa |
| `ci` | Perubahan pipeline CI/CD |

### 10.2.2 Scope

Scope mengacu pada modul atau domain:

| Scope | Contoh |
|-------|--------|
| `auth` | Modul autentikasi |
| `user` | Modul manajemen user |

# 11. Pull Request dan Workflow Rules

## 11.1 Aturan PR

- **Maksimal 1 feature per PR**. PR tidak boleh menggabungkan lebih dari satu fitur atau bugfix yang tidak terkait.
- PR harus memiliki branch source dari `develop` kecuali `hotfix`.
- PR harus melewati Automated Checks sebelum merge:
  - Lint dan format check lulus
  - Unit test lulus
  - Integration test lulus
  - Build berhasil
- PR harus memiliki deskripsi yang jelas berisi:
  - Tujuan perubahan
  - Dampak pada modul lain
  - Link ke task atau user story (PB-XXX, TASK-XXX, API-XXX)
  - Screenshot atau evidence test jika diperlukan
- PR tidak boleh berisi komentar debug, log debug, atau kode yang sengaja diberi komentar tanpa alasan
- Conflict harus diselesaikan sebelum merge; tidak boleh merge dengan konflik yang menyembunyikan perubahan

## 11.2 Code Review

- Minimal satu reviewer sebelum merge ke `develop`
- Reviewer memeriksa kepatuhan terhadap coding standard, naming convention, dan aturan keamanan
- Reviewer memastikan tidak ada hardcoded credential atau data sensitif
- Reviewer memastikan setiap endpoint memiliki validation dan test
- Reviewer memastikan error handling terstruktur

# 12. Defensive Programming Rules

- Semua pointer yang diterima dari eksternal WAJIB dicek nullability sebelum digunakan
- Struktur database atau API response WAJIB divalidasi sebelum dipetakan ke entity atau DTO
- Semua channel goroutine WAJIB memiliki select default atau timeout
- Semua context WAJIB dicek `ctx.Err()` sebelum lanjut operasi
- Semua channel WAJIB ditutup setelah selesai digunakan
- Menggunakan `sync.Once` atau mutex jika diperlukan untuk operasi concurrent
- Database connection diinisialisasi sekali dan dibagikan melalui dependency injection

---

# 13. Observability

- Setiap request dilengkapi span trace OpenTelemetry
- Handler mencatatkan method, path, dan role pengguna
- Repository mencatatkan query yang dijalankan dan durasinya
- Service mencatatkan nama use case dan statusnya
- Operasi async (import, analysis) mencatatkan status dari `PROSES`, `SELESAI`, hingga `GAGAL`
- Semua log dan trace disimpan dalam format yang dapat diagregasi
- Tidak ada log yang menampilkan data sensitif (password, token, email pribadi jika tidak perlu)

# 14. Teknis Implementasi Kendali

## 14.1 Dependensi

- Setiap package atau library baru harus disetujui sebelum ditambahkan
- Dependency diinfeksikan melalui `go.mod` atau `package.json`
- Tidak boleh ada dependency yang tidak digunakan (dead code dependency)

## 14.2 File dan Path

- Semua path file database, import, dan report harus disimpan sebagai relative path atau path terstruktur, bukan absolute path yang tergantung pada OS
- Nama file yang di-upload harus di-sanitize untuk mencegah path traversal

## 14.3 Database Transaction

- Operasi yang memerlukan konsistensi data (misal create analysis session beserta result) WAJIB menggunakan transaction
- Transaction di-rollback jika terjadi error pada salah satu operasi

## 14.4 Idempotensi

- Endpoint yang mengubah state (POST, PUT, DELETE) WAJIB mendukung idempotensi jika memungkinkan
- Import dan generate report WAJIB dapat diulang tanpa menimbulkan duplikasi data

# 16. Definition of Done (Code Level)

Sebuah task atau feature dianggap selesai jika:

1. Kode berformat sesuai coding standard
2. Nama variabel, fungsi, dan file sesuai naming convention
3. Semua endpoint memiliki validation sesuai API contract
4. Unit test ditulis dan lulus dengan coverage minimum 85%
5. Integration test ditulis dan lulus
6. Semua error mengembalikan format terstruktur sesuai standar
7. Logging ditambahkan pada business logic yang relevan
8. OpenTelemetry tracing ditambahkan pada request dan operasi async
9. Tidak ada hardcoded credential atau data sensitif
10. Dokumentasi teknis atau komentar kode ditambahkan jika diperlukan
11. Conventional commit message ditulis sesuai format
12. Traceability ke requirement source tercantum di dalam PR atau commit message
